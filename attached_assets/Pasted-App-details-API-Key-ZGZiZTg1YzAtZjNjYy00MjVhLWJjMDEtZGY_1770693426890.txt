Background Processing for Runa Orders
All purchase operations through the Runa integration are processed as background operations. The Fluz API Adapter provides two response modes for Runa:
Synchronous Mode
Add header X-Execution-Mode: sync to wait for the background operation to complete. The API call will wait until the background purchase operation completes and will return full operation results in the response. Useful for testing, but may time out for complex purchases.
Asynchronous Mode (Default)
Returns immediately with an operation reference ID while the purchase continues processing in the background. Check status later by querying the order endpoint with the reference ID. Recommended for production to avoid timeouts.
Both modes rely on background processing, but they differ in how the API responds to the client.
Idempotency Support
Both the original Runa API and Fluz’s implementation support idempotency keys to prevent duplicate operations. Use the header:
X-Idempotency-Key: your-unique-key

Background Processing for TangoCard Orders
All purchase operations through the TangoCard integration are processed as background operations. However, for compatibility with TangoCard’s API behavior, the Fluz API Adapter may return full results synchronously by default.
Note: Even when results appear synchronous, all purchases are still scheduled and tracked as background operations internally for improved reliability and better handling of retries.
